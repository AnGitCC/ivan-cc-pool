# 问卷调查平台设计方案

## 1. 目标与定位

本项目首版定位为一个面向讲师使用的通用问卷调查平台，重点服务于培训、授课、活动反馈等场景。讲师登录后台后，可以创建问卷、创建场次、发起匿名收集、查看统计结果，并导出明细、汇总和 PDF 报告。

首版的最高优先级不是功能堆叠，而是以下三点：

- 页面点击、切换、弹窗反馈要快
- 员工扫码填写时打开速度要快、访问成功率要高
- 整体界面要简洁、高级，风格参考 Apple 官网首页的视觉气质

## 2. 首版范围

首版纳入范围如下：

- 讲师账号注册、登录、退出
- 问卷列表查看
- 问卷手工新建
- 问卷 Excel 模板下载、上传导入、自动解析
- 问卷编辑
- 基于问卷创建多个场次
- 场次独立填写链接与二维码
- 匿名填写
- 场次统计展示
- 场次原始明细导出
- 场次统计汇总导出
- PDF 截图式报告导出
- 二维码图片下载

首版暂不纳入：

- 管理员后台
- 子账号或协作权限
- 访问密码
- 自动生成分析结论
- 高级文本分析
- 复杂题型，如矩阵、排序、题库系统
- 定时自动关闭场次
- 跨场次、跨问卷的平台级分析

## 3. 核心产品对象

首版在产品层面的核心对象只有两个：

- 问卷
- 场次

设计原则如下：

- 问卷负责定义结构
- 场次负责发布、收集、统计、导出
- 单次填写记录会存储在系统中，但不作为一级业务对象出现在主导航中
- 所有统计、导出、PDF 报告都以场次为中心展开

## 4. 产品主链路

首版主链路定义为：

讲师登录 -> 创建问卷 -> 创建场次 -> 展示二维码/分享链接 -> 员工匿名填写 -> 查看统计 -> 导出数据/PDF -> 手动关闭场次

设计和实现都围绕这条主链路优化，优先保证体验稳定和响应速度。

## 5. 后台信息架构

### 5.1 登录后默认首页

讲师登录后默认进入讲师大屏，也就是统计首页。

首页规则如下：

- 优先展示最近一个进行中的场次
- 如果没有进行中的场次，则展示最近一个已关闭场次
- 首页主体展示当前场次统计概览
- 首页提供切换场次能力
- 首页提供创建场次入口
- 首页提供进入场次列表入口
- 首页提供导出入口
- 首页提供投屏填写入口

### 5.2 问卷页

创建问卷和编辑问卷统一收口到问卷列表页中，不放在首页主视觉区域中。

问卷页承接以下操作：

- 查看问卷列表
- 手工新建问卷
- 下载 Excel 模板
- 上传 Excel 导入问卷
- 编辑已有问卷
- 删除空问卷

删除规则如下：

- 删除入口只出现在问卷列表页
- 仅允许删除空问卷
- 空问卷定义为：关联场次数为 0
- 只要该问卷已经创建过任意场次，就不允许删除
- 删除前必须进行二次确认
- 本次不做软删除，直接执行物理删除
- 不可删除的问卷不提供可执行删除入口

### 5.3 场次页

场次以问卷为归属对象。每个问卷可以创建多个场次。

场次页承接以下操作：

- 查看某问卷下的场次列表
- 创建场次
- 查看场次统计
- 查看场次填写链接和二维码
- 关闭场次
- 导出场次数据

## 6. 首页与场次状态规则

### 6.1 场次状态

首版场次状态只保留两种：

- 进行中
- 已关闭

状态流转如下：

- 新建场次后默认是进行中
- 讲师手动点击关闭场次后，状态变为已关闭
- 首版不引入自动结束时间
- 结束时间由讲师主动控制

### 6.2 关闭后的访问行为

场次关闭后：

- 原填写链接仍然可以访问
- 原二维码仍然可以扫码进入
- 但进入后只展示场次已结束提示页
- 用户不能再继续填写和提交

### 6.3 首页默认取数规则

首页默认展示场次时，按以下顺序取数：

1. 最近创建且仍处于进行中的场次
2. 若没有进行中的场次，则取最近关闭的场次

## 7. 投屏填写与二维码展示

首页不常驻展示二维码和填写链接，而是采用按钮加弹窗的方式。

### 7.1 交互方式

- 首页提供明显的投屏填写按钮
- 点击后弹出投屏弹窗
- 弹窗内展示当前场次名称
- 弹窗内展示填写链接
- 弹窗内展示二维码
- 支持复制填写链接
- 支持下载二维码图片
- 关闭弹窗后回到首页继续查看统计

### 7.2 使用场景

该设计主要服务于讲师现场投屏：

- 平时首页保持干净，以统计为主
- 需要让学员扫码时，再点击按钮展示二维码
- 讲师可以将二维码图片下载后，分享至微信群、PPT、海报等其他渠道

### 7.3 二维码生成规则

二维码生成时，必须使用员工在各类网络环境下都可访问的填写域名，而不是后台域名。

也就是说：

- 二维码中的链接必须指向填写端专用公网域名
- 不允许用讲师后台域名生成二维码
- 不允许使用仅适合后台管理访问的域名生成二维码

## 8. 问卷结构设计

### 8.1 结构分组

一份问卷由两大分组组成：

- 基础信息
- 正式题目

规则如下：

- 基础信息用于采集描述性字段
- 基础信息不参与计分和总分统计
- 正式题目用于统计和计分

### 8.2 首版题型

首版支持以下基础题型：

- 单选
- 多选
- 文本
- 评分

题型使用原则：

- 单选、多选、评分是主要统计题型
- 文本题用于原文收集和导出
- 文本题首版不做复杂语义分析

### 8.3 计分规则

正式题目可分为两类：

- 计分题：配置了选项分值
- 非计分题：没有配置分值

规则如下：

- 只有正式题目参与计分
- 基础信息永远不参与总分、均分、排名类统计
- 单选题得分等于所选项分值
- 多选题首版采用所选项分值求和
- 评分题本质上按带分值的选项题处理
- 文本题不计分

### 8.4 统计规则

所有统计都围绕场次进行，不做首版平台级总览。

场次统计包括：

- 回收量
- 计分汇总
- 单题平均分
- 分数分布
- 非计分题选项频次和占比
- 基础信息字段分布
- 文本题明细展示

## 9. Excel 导入与建卷方式

首版支持两种创建问卷方式：

- 页面手工新建
- Excel 导入

这两种方式最终都落到同一套问卷 schema 中，避免后续维护两套逻辑。

### 9.1 Excel 模板

平台提供统一模板下载。模板建议包含以下关键列：

- 所属分组
- 题目标题
- 题型
- 是否必填
- 选项列表
- 选项分值

解析约束如下：

- 所属分组只允许基础信息或正式题目
- 选项列表使用固定分隔符
- 选项分值与选项顺序一一对应
- 若分值为空，则该题按非计分题处理

### 9.2 导入后处理

Excel 上传并解析成功后：

- 自动生成问卷结构
- 进入问卷编辑页
- 在编辑页顶部展示明确成功提示
- 允许讲师继续调整题目、选项、分值、顺序和必填状态

因此，Excel 导入定位为快速建卷入口，而不是一次性不可编辑的黑盒流程。

导入成功提示规则如下：

- 文案需要明确告诉讲师“导入已经成功”
- 提示出现位置放在问卷编辑页顶部
- 提示语义偏业务反馈，不使用技术化表达
- 建议提示内容为：`Excel 问卷已成功导入，请继续检查并完善内容。`

## 10. 统计页设计

### 10.1 展示方式

统计页采用模块化卡片展示，而不是长列表堆叠。

建议支持以下模块：

- 场次基础信息
- 回收概览
- 计分汇总
- 非计分题频次统计
- 基础信息字段统计
- 文本题明细

### 10.2 模块显示控制

讲师可以自行选择统计页展示哪些模块。

交互规则如下：

- 页面提供显示设置按钮
- 点击后可勾选要显示的统计模块
- 勾选的模块显示
- 取消勾选的模块隐藏

该设计兼顾两种场景：

- 日常查看，希望信息完整
- 投屏展示，希望页面更简洁

### 10.3 图表类型可配置

讲师可以为大屏页中的单个统计模块自行选择图表展示方式。

首版支持以下图表类型：

- 环形饼图
- 横向柱状图
- 竖向柱状图

配置原则如下：

- 配置粒度以单个统计模块为单位
- 不要求整个页面统一使用同一种图表
- 讲师可根据题目内容选择更合适的展示方式
- 图表切换应即时生效，不能造成明显等待

示例：

- 单选题可展示为环形饼图
- 选项较多的题更适合横向柱状图
- 分数分布更适合竖向柱状图

### 10.4 导出与图表配置联动

PDF 报告默认跟随当前统计页的模块显示配置和图表类型配置。

也就是说：

- 讲师先在统计页决定要展示哪些内容
- 再决定每个模块使用哪种图表
- 再导出 PDF
- PDF 默认沿用当前页面的图表样式

这样可以避免额外设计一套独立的报告配置系统。

## 11. 导出设计

导出始终以场次为中心。

首版支持以下导出能力：

- 原始答卷明细导出
- 统计汇总结果导出
- PDF 截图式报告导出
- 单条填写记录导出

其中，单条记录导出不作为一级产品对象存在，而是作为场次明细列表中的附属操作。

### 11.1 原始明细导出

适合一条记录一行，便于后续人工分析或再加工。

### 11.2 统计汇总导出

适合导出每题频次、占比、均分、总分、分布等聚合数据。

### 11.3 PDF 报告导出

首版 PDF 定位为可汇报的统计快照，而不是自动撰写分析结论的智能报告。

建议默认包含：

- 场次基本信息
- 回收概览
- 计分汇总
- 重点题目统计图表
- 非计分题频次统计
- 文本题附录或摘要列表

## 12. 技术路线

### 12.1 总体方案

首版采用单体式 MVP 路线，优先保证开发效率和上线速度。

推荐组合：

- Next.js
- React
- TypeScript
- PostgreSQL
- Prisma
- Redis 可选

### 12.2 为什么采用这条路线

该项目同时包含：

- 讲师后台
- 匿名填写页
- 文件导入
- 统计展示
- 数据导出
- PDF 导出

单体式 Web 应用更适合首版快速交付，也更容易围绕性能做统一优化。

## 13. 部署与网络访问架构

### 13.1 部署要求

考虑国内网络环境限制，首版部署需满足以下约束：

- 应用部署在 Vercel
- 填写端访问能力重点通过 Cloudflare 保障
- 数据库使用 Supabase

### 13.2 域名拆分方案

从产品入口上，后台域名和填写域名拆分。

建议如下：

- 后台域名，例如 `console.xxx.com`
- 填写域名，例如 `join.xxx.com` 或 `survey.xxx.com`

设计原则如下：

- 讲师后台和员工填写入口分离
- 填写端域名重点做 Cloudflare 优化
- 后台域名保持清晰独立，避免与扫码链路相互影响

### 13.3 二维码域名要求

二维码和填写链接必须使用填写端专用域名。

规则如下：

- 二维码中的 URL 一律使用填写域名
- 讲师后台复制的填写链接也一律使用填写域名
- 不允许后台域名混入员工扫码路径

## 14. 性能优先策略

速度是首版最高优先级要求，所有技术实现都必须围绕这一点展开。

### 14.1 用户感知目标

- 页面切换快
- 按钮点击反馈快
- 弹窗打开快
- 场次切换快
- 统计查看快
- 扫码打开填写页快

### 14.2 实现原则

- 首页优先展示摘要统计，再补充图表
- 场次切换先切关键数字，再更新详细内容
- 统计采用场次级预聚合或增量聚合，避免每次全量重算
- 问卷编辑以本地状态编辑为主，保存时统一提交
- 导出和 PDF 使用后台异步任务，不阻塞页面操作
- 填写页资源尽量轻量，优先保证移动端扫码打开速度
- 对最近访问场次的统计结果做短时缓存

### 14.3 性能优先级排序

首版性能优先级建议如下：

1. 员工扫码后填写页打开速度
2. 首页和场次统计页的切换速度
3. 按钮、弹窗、勾选等前端交互反馈速度
4. 导出任务的生成速度

## 15. 视觉设计方向

### 15.1 整体风格

整体风格参考 Apple 官网首页的视觉气质，但不直接复制具体页面布局。

需要借鉴的是：

- 大留白
- 强层级
- 克制排版
- 高级感
- 轻盈感
- 秩序感

### 15.2 页面气质

后台虽然是业务系统，但关键页面需要更像高质感产品界面，而不是传统厚重后台。

落地原则：

- 页面信息分层清晰
- 控件简洁
- 重点操作明确
- 不堆砌边框和颜色
- 图表和卡片保持克制

### 15.3 品牌配色

主题色使用公司品牌绿色，要求是鲜艳但柔和。

配色原则：

- 主背景以白色、浅灰、中性色为主
- 绿色主要用于主按钮、选中态、关键高亮和成功状态
- 避免整页大面积使用高饱和绿色

### 15.4 重点页面表现

首页大屏建议：

- 大标题
- 大数字
- 清晰的关键操作按钮
- 克制的图表视觉

二维码弹窗建议：

- 白底
- 居中布局
- 大尺寸二维码
- 清晰的复制链接和下载二维码按钮

### 15.5 后台底部标识

讲师后台所有页面的最底端需要固定保留一行居中文案：

`人力资源本部 · 人才发展部`

使用原则如下：

- 仅在讲师后台页面显示
- 员工匿名填写页不显示
- 位置统一放在页面底部居中
- 视觉风格保持克制，不喧宾夺主

## 16. 风险与约束

首版关键风险如下：

- Excel 模板若不够严格，导入解析容易失败
- 手工建卷与 Excel 导入若不共用 schema，统计会出现不一致
- PDF 截图式导出要控制分页和图表显示稳定性
- 场次关闭后必须给出已结束提示页，不能直接出现 404
- 投屏场景对视觉层次和交互简单性要求更高
- 国内网络环境下，填写域名和二维码链路必须优先验证访问稳定性

## 17. 首版结论

首版方案正式收敛如下：

- 采用单体式 MVP 路线
- 讲师登录后默认进入讲师大屏
- 首页优先展示最近一个进行中的场次
- 问卷创建支持 Excel 导入和页面手工新建
- 统计和导出全部围绕场次展开
- 首页通过弹窗展示二维码和填写链接
- 弹窗支持复制链接和下载二维码图片
- 大屏页支持为单个统计模块切换环形饼图、横向柱状图、竖向柱状图
- 二维码必须使用填写端专用公网域名
- 填写入口和后台入口拆分为不同域名
- 数据库使用 Supabase
- 性能和访问速度为首版最高优先级
- 视觉风格参考 Apple 官网首页，品牌主色为鲜艳但柔和的绿色
- 讲师后台所有页面底部固定居中显示 `人力资源本部 · 人才发展部`
